## 为什么先从 V4L2 开始

在 Linux 环境下，视频相关设备通常不会直接暴露给业务层随意访问，而是通过统一框架进行抽象。

V4L2（Video for Linux 2）就是这套抽象的核心。它解决的不是“某一个摄像头怎么读数据”，而是：

- 如何统一描述视频设备能力
- 如何让用户态通过一致接口访问不同厂商设备
- 如何把采集、格式协商、buffer 管理、流开关等流程组织起来
- 如何让驱动和上层应用之间形成清晰边界

所以理解视频子系统，往往就是从理解 V4L2 开始。

## 一个典型的视频采集链路

从系统角度看，一条视频链路通常可以抽象成下面几层：

1. 图像输入源
   - 例如 sensor、摄像头模组
2. 总线或控制接口
   - 例如 I2C、MIPI CSI、并口接口
3. 中间处理模块
   - ISP、缩放、格式转换、裁剪等
4. 采集设备节点
   - 最终以 `/dev/video*` 的形式暴露给用户空间
5. 用户态应用
   - 例如 ffmpeg、gstreamer、自研采集程序

如果只看应用层，往往只会觉得“打开 `/dev/video0`，然后读帧”。
但从驱动角度看，真正复杂的地方在于：

- sensor 是否正常出流
- 链路时序是否匹配
- 像素格式是否协商成功
- buffer 是否正确分配和回收
- 中间模块是否被正确配置

## V4L2 驱动通常负责什么

一个最基本的视频驱动，通常需要处理几类核心能力：

### 1. 设备注册

驱动需要把自己注册到 V4L2 框架中，并创建对应的视频设备节点。

这一步解决的是“系统如何知道这是一个可用的视频设备”。

### 2. 格式协商

上层应用通常会设置：

- 分辨率
- 像素格式
- 帧率

驱动需要判断这些参数是否被硬件支持，并返回最终可用配置。

### 3. Buffer 管理

视频采集本质上是连续数据流，不可能每来一帧临时申请一次内存，所以一般都会提前准备 buffer 队列。

这里通常涉及：

- buffer 分配
- queue / dequeue
- DMA 映射
- cache 一致性

这是性能和稳定性问题最集中的区域之一。

### 4. Stream on / off

真正开始采集前，需要完成一系列准备动作：

- 配置硬件寄存器
- 打开 sensor 输出
- 启动 DMA
- 使能中断

停止采集时则要把这些状态有序释放。

### 5. 错误处理与状态恢复

实际项目里，最难的往往不是“跑通一次”，而是：

- 热插拔后是否还能恢复
- sensor 异常时如何处理
- 超时、丢帧、格式错误如何定位
- 多模块联动时状态是否一致

## 学视频子系统时建议优先抓住什么

如果一开始就陷入大量接口和回调，很容易迷失。更有效的方法是优先抓住下面四条主线：

### 主线 1：数据从哪里来，到哪里去

先画出链路图，明确：

- 输入源是谁
- 数据经过哪些模块
- 最后由哪个 video node 输出

### 主线 2：配置是在什么时候完成的

例如：

- probe 阶段做了什么
- open 时做了什么
- set format 时做了什么
- stream on 时做了什么

### 主线 3：buffer 生命周期

把每个 buffer 的状态变化梳理清楚，很多 bug 就会变得容易定位。

### 主线 4：问题如何被观测

要建立最基本的调试视角，例如：

- 当前设备节点是否存在
- 当前支持哪些 format
- stream on 到底有没有成功
- 中断有没有进来
- 有没有 frame done

## 小结

视频子系统的难点，不在于接口多，而在于它天然是一个 **跨模块、跨层次、强状态依赖** 的系统。

只要先抓住链路、状态、buffer 和观测点这几个核心维度，后续无论看驱动代码、排查问题还是做模块设计，都会清晰很多。
